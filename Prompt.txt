Diseña una aplicación web SaaS para restaurantes con control de mesas, pedidos, reservas, meseros, inventario con kardex, facturación electrónica del SRI (Ecuador), integración con POS y multiempresa.

Tecnologías
Frontend: React 18 + TypeScript + Vite.

Estilos: Tailwind CSS (diseño responsive móvil/tablet/escritorio).

Iconos: Lucide React.

Backend/BD: Supabase (PostgreSQL) con Row Level Security (RLS) para multiempresa.

Deploy: Vercel (frontend) + Supabase.

Multiempresa, usuarios y RLS
Tablas principales con columna empresa_id para aislar los datos por empresa (tenant).
​

Roles de usuario:

Admin plataforma: crea empresas, asigna planes, crea usuario Oficina inicial.

Oficina: pertenece a una empresa, puede configurar productos, mesas, facturar, ver reportes, cierres de caja.

Mesero: puede ver solo sus mesas/pedidos, tomar pedidos, enviar a cocina, pasar a facturación.

Implementar RLS en Supabase:

Política general: cada usuario solo ve/modifica filas donde empresa_id coincide con la empresa del usuario.

Incluir en el JWT un claim empresa_id y el rol interno (admin/oficina/mesero) para las políticas.
​

Tablas base sugeridas: empresas, usuarios, roles, mesas, reservas, clientes, productos, inventarios, kardex, pedidos, pedido_detalles, comprobantes, comprobante_detalles, formas_pago, pagos, parametros_sri, planes, empresa_planes.

Inventario y kardex
Tabla productos:

Campos: nombre, categoría, precio_venta (con IVA incluido), controla_inventario (bool), stock_minimo, activo, indicador de IVA, etc.

Tabla inventarios:

producto_id, stock_actual, empresa_id.

Tabla kardex:

Movimiento de inventario: fecha, tipo_movimiento (compra, venta, ajuste, devolución), producto_id, cantidad, costo_unitario, costo_total, referencia (número de documento).

Lógica:

Al facturar un producto con controla_inventario = true, registrar movimiento de salida en kardex y actualizar stock_actual.

Añadir alerta si stock_actual <= stock_minimo para mostrar en panel Oficina.

Formas de pago e integración POS
Tabla formas_pago:

Ejemplos: efectivo, tarjeta crédito, tarjeta débito, transferencia, otros.

Tabla pagos:

Vinculada a comprobantes, permite múltiples formas de pago por comprobante (mix efectivo/tarjeta).

Campos: forma_pago_id, monto, referencia (número de voucher, autorizador del POS, etc.).

Flujo:

En la pantalla de facturación, permitir seleccionar una o varias formas de pago, registrar montos y referencias.

Dejar la integración con POS como capa adicional (por ejemplo, función que reciba confirmación de pago desde POS o pasarela y actualice el estado de pago).

Reportes y monetización
Reportes:

Monetización:

RLS y seguridad
Activar RLS en todas las tablas sensibles.

Usar políticas tipo:

USING (empresa_id = auth.jwt() ->> 'empresa_id') para SELECT/INSERT/UPDATE/DELETE.

Excepciones para el rol Admin plataforma, que puede ver todas las empresas.

Asegurar que los triggers de inventario y kardex respeten el empresa_id.
